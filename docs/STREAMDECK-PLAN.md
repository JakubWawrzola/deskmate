# Stream Deck integration — plan and status

Goal: control Home Assistant from Elgato Stream Deck keys.

## Status (2026-07-14): MVP IMPLEMENTED

The plugin exists in `streamdeck-plugin/` (README with build/install
instructions and a manual testing checklist). It builds (`npm run build`
+ `npx tsc --noEmit` green). NOT YET TESTED on physical hardware.

### Architecture change relative to the original plan

The original plan (below, section "Target architecture — via-Deskmate
variant") assumed the plugin would talk to a local Deskmate instance over WS.
Decision made on 2026-07-14: **the plugin is standalone and connects DIRECTLY
to HA**:

- actions: REST `POST {base}/api/services/{domain}/{service}` (Bearer token),
- live entity state: WebSocket `{base}/api/websocket` (auth + subscribe to
  `state_changed` + `get_states`, cache limited to tracked entities, reconnect
  with backoff, ping/pong),
- configuration: Base URL + long-lived token in the plugin's global settings
  (entered in the Property Inspector; zero hardcoding, token never enters the
  repo).

Reason: works without Deskmate running, no extra localhost channel to secure,
simpler release cycle. The via-Deskmate variant remains a possible future
option (entity dropdown backed by the cache).

### Implemented (v0.1.0)

- [x] **Toggle Entity** action — `homeassistant.toggle`, the key shows live
      on/off state (manifest with 2 states + `DisableAutomaticStates`,
      `setState` driven by the WS subscription).
- [x] **Call Service** action — `domain.service` + optional entity_id +
      optional `data` (JSON, validated before sending).
- [x] **Activate Scene** action — `scene.turn_on`.
- [x] HA client (singleton): REST + WS, reconnect (2s -> 30s), ping/pong,
      stops after `auth_invalid` (doesn't spam HA with a bad token).
- [x] Property Inspector (sdpi-components from the Elgato CDN) with global
      URL/token fields.
- [x] Resilience: missing config / network error / bad JSON -> `showAlert()`
      on the key, errors logged via `streamDeck.logger`, plugin doesn't crash.
- [x] SDK v2 manifest (Node 20), UUID `com.homeos.homeassistant`,
      placeholder PNG icons generated by a script (no network needed).
- [x] Build: rollup + TS (layout based on the official
      streamdeck-plugin-template), `ws` bundled (optional native modules have
      a JS fallback).

## Archive: original assessment and the via-Deskmate variant (2026-07-10)

It's doable, but it's a separate, non-trivial module. It requires the Elgato
SDK (the plugin is a separate Node/WebSocket process managed by the Stream
Deck app), its own manifest and release cycle. It doesn't block any other
Deskmate feature.

### Target architecture — via-Deskmate variant (NOT implemented)

```
Stream Deck app ── launches ──> deskmate-streamdeck plugin (Node, SD SDK v2)
                                    │ localhost WS (configurable port)
                                    ▼
                              Deskmate (Rust core)
                                    │ MQTT (existing connection)
                                    ▼
                              Home Assistant
```

Advantages: a single configuration (broker in Deskmate), no second channel to
secure. Disadvantages: requires Deskmate to be running and a new WS server in
the Rust core. Steps and risks for this variant — see this file's git history.
